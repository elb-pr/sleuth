# Database Usage Guide

All investigation data is stored in **CSDb (Claude Sleuth Database)** — a Cloudflare Workers MCP server backed by D1 (SQLite). CSDb is the sole persistence layer: there are no local JSON files to maintain. Data is written to CSDb as it is produced and persists across sessions.

---

## Data Model

All data is scoped to an `investigation_id` obtained from `CSDb:create_investigation` at case start. The schema enforces referential integrity at write time — malformed records are rejected, not silently accepted.

### POLE Entities

Persons, organisations, locations, events, objects, and other entity types identified during investigations. Each entity carries:

- `name` — primary name or title
- `type` — person, organisation, location, event, object, domain, email, phone, account, document, vehicle, address
- `aliases` — alternative names, trading names, maiden names
- `identifiers` — passport, company number, LEI, DUNS, tax ID, custom
- `attributes` — birth date, nationality, jurisdiction, status, etc.
- `source` — provenance of this record
- `source_grade` — Admiralty 6x6 grade (e.g. A1)
- `notes` — free-text analyst notes

**Tools:** `add_entity`, `search_entities` (query + type filter), `update_entity`

### Relationships

Directed and undirected links between entities. Each relationship carries: source and target entity IDs, type (e.g. `director_of`, `shareholder_of`, `communicated_with`), direction, confidence weight, source grade, and notes.

**Tools:** `add_relationship`, `get_relationships` (by entity ID)

### Timeline Events

Chronological records with datetime, description, category (communication, movement, financial, meeting, incident, document, other), involved entities, location, and source.

**Tools:** `add_timeline_event`, `get_timeline` (filterable by category and entity)

### Geo-Intelligence

Point locations associated with entities. Each location carries a label, coordinates, entity reference, observation timestamp, source, and source grade.

**Tools:** `add_location`

### Evidence Metadata

Registry of preserved evidence artefacts. Each entry carries a description, artefact type, SHA-256 hash, URL, and storage location.

**Tools:** `register_evidence`

### Source Grades

Admiralty 6x6 grades recorded per claim. Each grade carries the source name, reliability (A-F), credibility (1-6), the claim text, and justification.

**Tools:** `record_grade`

### Investigation Notebook

Free-form markdown content persisted directly — no script intermediary. Captures analytical position, unresolved questions, and decisions.

**Tools:** `save_notebook`, `load_notebook`

### Task Progress

Task runner state: current task index and list of completed task IDs.

**Tools:** `save_progress`, `load_progress`

---

## Maintenance Rules

1. **Every entity must have a source and source grade.** No record enters the database without Admiralty 6x6 grading.
2. **IDs are generated by CSDb.** Do not fabricate entity IDs — use the ID returned by `add_entity`.
3. **Write as you go.** Entities, relationships, evidence, grades, and locations are written to CSDb as they are produced — do not batch to session end.
4. **Notebook and progress are checkpointed.** Save progress after every `done` command. Save the notebook at phase transitions and whenever a finding materially shifts the picture.

---

## Session Protocol

### Session start

1. Call `CSDb:list_investigations` to identify the active case
2. Call `CSDb:load_progress` with the `investigation_id` to restore task runner state
3. Call `CSDb:load_notebook` to restore the Investigation Notebook
4. Resume from the current task position

### Session end

No special action required if write-through discipline was maintained. If in doubt, call `CSDb:save_progress` and `CSDb:save_notebook` as a final checkpoint.

---

## Investigation Lifecycle

### Starting a case

Call `CSDb:create_investigation` with a descriptive case name. The returned `investigation_id` scopes all subsequent calls.

### During a case

Use `CSDb:get_statistics` for a dashboard view of entity counts, relationship counts, timeline events, evidence items, and grades.

Use `CSDb:load_investigation` for a full state dump — useful for feeding `network_graph.py` or `chronological_matrix.py` with complete data.

### Closing a case

1. Verify all 56 tasks complete via `task_runner.py status`
2. Call `CSDb:get_statistics` and record counts in the Case Summary Record
3. Call `CSDb:close_investigation` to mark the case as closed
4. Closed investigations remain queryable via `CSDb:load_investigation` for future cross-case reference

### Multiple investigations

CSDb supports multiple concurrent investigations. Each maintains fully isolated entities, relationships, timeline, evidence, grades, notebook, and progress. Use `CSDb:list_investigations` to see all cases.

---

## Known Limitations

- **No delete operations.** Entities and relationships cannot currently be removed once created. If a record is created in error, document the correction in the notebook and update the entity with a note. Delete tools are planned.
- **No cross-investigation queries.** Each investigation is fully isolated. Cross-case link analysis requires loading multiple investigations and comparing manually.
