# t5.1 — Digital Evidence Hashing

**Tool:** `evidence_preservation.py`

## Procedure

1. YOU MUST process every digital file, screenshot, or artefact through SHA-256 hashing at the moment of capture. Any delay between capture and hashing is FORBIDDEN.

2. You are REQUIRED to run `evidence_preservation.py` to compute the SHA-256 hash. It is CRITICAL that the tool is operational and tested before any collection begins — confirmed in t3.4a. Using an untested hashing tool is FORBIDDEN.

3. ALWAYS record the hash value as a mandatory metadata field in `templates/database/evidence-register.md` alongside: unique reference number, filename, file size, format, and UTC timestamp of capture. An evidence entry without a hash is FORBIDDEN.

4. YOU MUST store the hash log separately from the evidence files. The hash log and the evidence files MUST NOT reside in the same writable location — co-located storage is a complete systematic failure.

5. ALWAYS re-hash on every transfer between storage systems. YOU MUST compare the new hash against the original — any discrepancy triggers an immediate integrity investigation.

6. At the Phase 2 and Phase 6 Checkpoints, you are REQUIRED to perform a full integrity sweep: re-hash all Evidence Register items and verify against recorded values. Any discrepancy is an integrity failure and MUST be escalated immediately.

7. It is CRITICAL that you understand: hashing proves integrity, not authenticity. A fabricated document produces a valid hash — authentication MUST follow the verification steps in t3.3b. Format conversion changes the hash — YOU MUST preserve original formats and hash every format variant separately.
