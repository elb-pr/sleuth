# t6.3 — Deterministic Matching for Unique Identifiers

**Tool:** CSDb MCP

## Procedure

1. YOU MUST designate deterministic keys for each entity type before extraction begins:
   - **Person:** Passport number, national ID, tax ID (NIN/SSN), driver's licence number.
   - **Organisation:** Company registration number, DUNS number, LEI, VAT number.
   - **Object:** VIN, IMEI, cryptocurrency wallet address.
   - **Account:** Bank account number + sort code/SWIFT, email address, phone number, platform-specific social media account ID.

2. ALWAYS compare deterministic key fields of every new record against all existing Entity Register records:
   - **Exact match found:** Append new source attribution, date, and field data to the existing record. Creating a duplicate is FORBIDDEN.
   - **No match found:** Create a new entity record and index its deterministic keys for future matching.
   - **Conflicting match:** Same deterministic key, conflicting attributes — YOU MUST flag immediately for manual review. Auto-merging conflicting records is FORBIDDEN. Possible explanations (data entry error, identity fraud, document reuse) are analytically significant.

3. It is CRITICAL that deterministic matching takes ABSOLUTE PRECEDENCE over probabilistic methods (t7.2). Even if probabilistic algorithms suggest a merge based on name/address similarity, a mismatch on a deterministic key MUST block the merge.

4. When no deterministic keys are available, YOU MUST route to the probabilistic pipeline (t7.2) and record the absence of deterministic keys as metadata — it signals reduced identity certainty.

5. ALWAYS log every deterministic merge: which key triggered it, which source records were combined, and which fields were contributed by each source. Pre-merge records MUST be preserved in audit history — erroneous merges MUST be reversible without data loss.

6. YOU MUST log all conflicts detected during matching in a separate Conflict Log with a priority flag proportionate to the investigative significance of the affected entity.
