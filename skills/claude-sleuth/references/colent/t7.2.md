# t7.2 — Probabilistic Matching and Human Review

**Tool:** CSDb MCP

## Procedure

1. When no deterministic key triggers an automatic merge, YOU MUST apply attribute-level comparison across multiple fields simultaneously:
   - **Name:** Exact string, Jaro-Winkler (transpositions), Soundex/Metaphone (phonetic), transliteration normalisation.
   - **Date of Birth:** YOU MUST check both DD/MM/YYYY and MM/DD/YYYY orderings — format-swap is the most common discrepancy. Partial matches (year + month only) receive reduced weight.
   - **Address:** Normalise abbreviations (St./Street, Rd./Road, Apt./Apartment), then compare by string similarity. Same-postcode proximity receives a lower but non-zero weight.
   - **Behavioural:** Writing style, posting times, shared network connections, device fingerprints may supplement demographic matching in digital investigations.

2. ALWAYS weight matches by attribute rarity. Agreement on a common name in a UK dataset is far less diagnostic than agreement on an unusual surname. You are REQUIRED to reference population frequency tables to weight name matches accordingly. A common DOB (e.g., 01/01/xxxx — a frequent placeholder in low-quality datasets) MUST carry reduced weight.

3. For candidate pairs (scores between auto-merge and auto-reject thresholds), YOU MUST present the analyst with a structured Match Summary Card: agreeing fields highlighted, disagreeing fields highlighted, individual weights, composite score, and links to original source documents for both records.

4. Analyst decisions MUST be one of: **Confirm** (merge, record analyst identity), **Reject** (record reason, prevent re-presentation), or **Defer** (generate a new IR to seek disambiguating evidence). Unrecorded decisions are FORBIDDEN.

5. Must-Not-Match rules MUST be implemented as explicit exclusions for pairs an analyst has determined are distinct entities, regardless of future attribute similarity. It is CRITICAL that these rules persist — failing to enforce them allows false positives to re-emerge as new data arrives.

6. ALWAYS preserve pre-merge records in audit history. YOU MUST record for every confirmed merge: agreeing attributes, assigned weights, composite score, approving analyst. Record for every rejected pair: reason for rejection. All probabilistic merges MUST be reversible.
