# Decisions

### DEC-001: Sub-skill architecture for reference folders
**Date:** 2026-03-30
**Context:** The four references/ subdirectories contain cross-cutting content that resists clean folder separation. Files belong in multiple states simultaneously (e.g. network.md is both a thinking workflow and an analysis tool task). A simple "consolidate each folder independently" approach kept falling apart because the folders are entangled.
**Decision:** Treat each references/ folder as its own sub-skill. Each folder gets a methodology.md that IS a skill file (without the YAML header) — with identity, constraints, methodology, reference routing, examples, and constraints reminder. The parent SKILL.md becomes a router/kernel dispatching to these sub-skills based on investigation state. tools/ remains a browsable shelf, not a sub-skill.
**Rationale:** This solves the entanglement problem by giving each folder a clear domain boundary enforced by the sub-skill's identity and constraints. Files that span states are owned by one sub-skill and referenced by others via state transitions. The SAVVY methodology applies directly to writing each methodology.md, giving structural consistency. The pattern also matches how the investigation workflow actually operates — as a state machine where you can move between research, analysis, and thinking freely.
**Alternatives considered:** (1) Consolidate each folder independently as a flat reference collection — rejected because cross-cutting content has no clear home without a state machine framing. (2) Merge all references into one flat folder with a single master file — rejected because it would be too large and lose the state-based routing that makes the skill usable. (3) Build the templates first and derive reference structure from template needs — rejected because templates depend on stable references, not the other way round.
