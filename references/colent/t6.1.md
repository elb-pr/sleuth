Task 6.1, **Implementing the POLE Schema**, is the structural prerequisite for all subsequent data ingestion. Before any raw intelligence is parsed or extracted, the investigator must map the investigation's data landscape to the operational ontology established in Phase 1 (Task 2.5). The **POLE** model — **Person, Object, Location, Event** — is the canonical framework used across UK law enforcement, allied intelligence services, and an increasing number of OSINT practitioners to categorise all intelligence into four discrete entity types.

The detailed procedural breakdown of this task is as follows:

### 1. Defining Entity Types and Their Schema Keys
Each of the four POLE categories must be configured with mandatory and optional fields before extraction begins. Retrospective schema changes are costly and risk data loss.
*   **Person:** Minimum required fields include `givenName`, `familyName`, `birthDate`, `nationality`, and `gender`. Standard optional fields include aliases (`additionalName`), historical addresses, employment history, telephone numbers, email addresses, social media handles, physical description, and identification documents (passport numbers, national ID, DUNS numbers).
*   **Object:** Covers physical and digital assets — vehicles (registration, VIN, make, model), financial accounts (account type, identifier, platform), communication devices (IMEI, phone number), documents (type, hash value, archive URL), and increasingly, cryptocurrency wallet addresses and dark web monikers.
*   **Location:** Minimum fields include address or geographic coordinates and type classification (residential, commercial, institutional). Optional fields include grid references, historical names, geospatial polygons, and ownership/tenancy records.
*   **Event:** Covers discrete occurrences with temporal and spatial anchors — meetings, transactions, communications, sightings. Minimum fields include event type, date/time (UTC-normalised), and location reference. Optional fields include participant list, description, and associated entity references.

### 2. Defining Relationship (Link) Types
Beyond entity types, the schema must predefine the vocabulary of relationships that will connect entities in the database.
*   **Standard Link Types:** Investigative best practice keeps link types minimal and meaningful. The standard vocabulary includes: `ASSOCIATED_WITH`, `OWNS`, `EMPLOYS`, `COMMUNICATES_WITH`, `FAMILY_OF`, `DIRECTOR_OF`, `LOCATED_AT`, `PARTY_TO`, and `WITNESSED_BY`.
*   **Link Metadata:** Every relationship must carry its own metadata: source attribution, date range (when the relationship was active), strength/confidence indicator, and analyst notes. Over-creating bespoke link types is a documented failure mode in platforms like i2 iBase — it fragments the schema and makes cross-case analysis impossible.
*   **Directionality:** Links must be directional where the relationship is asymmetric. "Entity A employs Entity B" is not the same as "Entity B employs Entity A."

### 3. Schema Extensibility for Novel Data Types
Rigid schemas fail when new data types emerge mid-investigation (e.g., cryptocurrency wallets, encrypted messaging identifiers, NFT ownership).
*   **Ad-Hoc Property Fields:** The schema should reserve generic extensibility mechanisms — a `Notes` or `customProperties` field — to capture data that does not fit the predefined structure without forcing a schema redesign.
*   **Semantic Typing:** In enterprise platforms like Palantir Gotham or IBM i2, semantic types allow subcategorisation within broad entity types. For example, `Person` may carry semantic subtypes of `Offender`, `Witness`, `Subscriber`, or `Beneficial Owner`, each inheriting the core Person fields while adding type-specific properties.

### 4. Alignment with Intelligence Requirements
The schema is not designed in a vacuum — it must directly serve the **Intelligence Requirements (IRs)** defined in Phase 1 (Task 2.1).
*   **IR-to-Schema Mapping:** Each IR should map to specific entity types and fields. If an IR asks "What properties does Entity A own in Jurisdiction X?", the schema must include Location entities with ownership fields linked to Person or Organisation entities.
*   **Preventing Over-Collection:** If a data point does not populate any field relevant to any IR, it should not be extracted. This prevents the creation of "orphan records" that degrade database usefulness without contributing to the investigation's goals.

### 5. Quality Gate Before Extraction
The schema must be reviewed and approved before any Task 6.2 parsing begins.
*   **Schema Sign-Off:** The lead investigator or case supervisor confirms that the entity types, link types, and mandatory fields are sufficient to address all active IRs.
*   **Downstream Compatibility:** The schema must be verified against the downstream tools that will consume it — the **Network Architecture Template** (Phase 4, Task 10.1) requires typed edges; the **Chronological Matrix** (Phase 4, Task 9.1) requires UTC-normalised event records. If the schema cannot produce these outputs, it must be revised before extraction begins.
