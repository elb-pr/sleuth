Task 6.2, **Parsing Unstructured Text**, is the transformation step where raw investigative material — witness statements, scraped web data, leaked documents, surveillance logs, or financial filings — is converted into structured records conforming to the POLE schema defined in Task 6.1. This task is executed through the **Entity Extraction & Profiling Template** and represents the boundary between human-readable narrative and machine-readable intelligence.

The detailed procedural breakdown of this task is as follows:

### 1. Input Classification
Before parsing begins, the investigator must classify the nature of the incoming data to select the appropriate extraction strategy.
*   **Structured Inputs:** Data already in tabular or semi-structured format (CSV exports from databases, company registry filings, financial spreadsheets). These require field mapping rather than entity extraction — aligning the source's column headers to the POLE schema keys.
*   **Semi-Structured Inputs:** Data with some inherent structure but embedded in narrative (web pages with predictable layouts, social media profiles with standard fields, PDF forms). These require template-based extraction targeting predictable field locations.
*   **Unstructured Inputs:** Free-text narrative with no predictable structure (witness statements, intercepted communications, investigative notes, news articles). These require Named Entity Recognition (NER) techniques to identify and extract persons, organisations, locations, dates, and other entities from continuous prose.

### 2. Named Entity Recognition (NER) for Unstructured Text
When processing free-text inputs, the extraction pipeline must identify and classify entity mentions within the narrative.
*   **Core NER Categories:** The extraction must identify at minimum: person names, organisation names, locations (addresses and place names), dates and times, monetary values, identification numbers (passport, company registration, phone numbers, email addresses), and vehicle identifiers.
*   **Tooling:** Production NER is performed using libraries such as **spaCy** (fast, production-grade, supports custom entity types), **Hugging Face Transformers** (local models for NER without API calls), or **Stanza** (Stanford NLP, supporting 70+ languages). For investigations spanning multiple languages, multilingual NER models are essential.
*   **Coreference Resolution:** NER alone identifies entity mentions, but a single entity may be referenced multiple ways within a document ("John Smith", "Mr Smith", "the defendant", "he"). Coreference resolution links these mentions to a single entity record, preventing the creation of duplicate entries.

### 3. Schema-Conformant Record Creation
Extracted entities must be mapped to the predefined POLE schema, not stored as raw text extracts.
*   **Field Population:** Each extracted entity generates a structured record with its schema-mandated fields populated. A person mention generates a Person record with `givenName`, `familyName`, and any other identifiable attributes. Missing fields are left explicitly blank — not guessed or inferred.
*   **Relationship Extraction:** Beyond entities, the parser must identify relationships expressed in the text. Phrases like "director of", "employed by", "travelled to", or "transferred funds to" generate typed links between the relevant entity records, using the link vocabulary defined in Task 6.1.
*   **Temporal Anchoring:** Where dates or time references are associated with entities or relationships, they must be normalised to UTC and attached to the relevant record as `dateObserved` or `dateRange` metadata.

### 4. Provenance Attachment
Every record created during parsing must carry its source provenance, linking it back to the specific document and passage from which it was extracted.
*   **Source Reference:** The Source Register identifier (from Task 5.3) of the document being parsed.
*   **Passage Location:** The specific page number, paragraph, line number, or character offset within the source document where the entity or relationship was found. This enables a third party to verify the extraction against the original text.
*   **Extraction Method:** Whether the record was created by manual analyst extraction, automated NER, or a combination. Automated extractions must be flagged for human review before they are treated as verified records.

### 5. Quality Controls
Parsing unstructured text is inherently error-prone. The following controls prevent garbage data from entering the Entity Register:
*   **Analyst Review Gate:** All automatically extracted records must be reviewed by a human analyst before being committed to the Entity Register. This is particularly critical for ambiguous extractions — names that could refer to multiple individuals, locations that are common across jurisdictions, or dates that are expressed ambiguously.
*   **Confidence Tagging:** Each extracted record receives an extraction confidence score — High (unambiguous extraction with clear context), Medium (plausible extraction requiring corroboration), or Low (ambiguous extraction that may be incorrect). Low-confidence extractions are quarantined for review rather than ingested directly.
*   **Deduplication Check:** Before a new record is committed, it is checked against existing records in the Entity Register using the deterministic matching rules defined in Task 6.3. If a match is found, the new data is merged with the existing record rather than creating a duplicate.
