Task 6.3, **Deterministic Matching for Unique Identifiers**, is the first line of defence against record duplication. Before any probabilistic or fuzzy matching is attempted, the system enforces exact-match rules on high-stakes identifiers — data points that are globally unique and definitively link a record to a specific real-world entity. If two records share the same passport number, they refer to the same person. No probability calculation is required.

The detailed procedural breakdown of this task is as follows:

### 1. Identifying Deterministic Keys
Not all data fields are suitable for deterministic matching. The investigator must designate specific fields as "deterministic keys" based on their uniqueness and reliability.
*   **Person Identifiers:** Passport number, national identity number, tax identification number (e.g., UK National Insurance Number, US SSN), driver's licence number, biometric identifiers.
*   **Organisation Identifiers:** Company registration number (e.g., Companies House number), Dun & Bradstreet DUNS number, Legal Entity Identifier (LEI), tax registration number (e.g., VAT number).
*   **Object Identifiers:** Vehicle Identification Number (VIN), IMEI number (mobile devices), cryptocurrency wallet addresses, domain registration WHOIS data.
*   **Account Identifiers:** Bank account numbers (combined with sort code or SWIFT/BIC), email addresses, phone numbers, social media account IDs (platform-specific unique identifiers, not display names).

### 2. The Matching Process
When a new record is extracted (via Task 6.2), its deterministic key fields are compared against all existing records in the Entity Register.
*   **Exact Match Found:** If an incoming record shares an identical deterministic key with an existing record, the system automatically links the new data to the existing entity. The new source attribution, date observed, and any additional field data are appended to the existing record without creating a duplicate.
*   **No Match Found:** The record is created as a new entity in the Register. Its deterministic keys are indexed for future matching.
*   **Conflicting Match:** If an incoming record shares a deterministic key with an existing record but other fields conflict (e.g., same passport number but different name), this is flagged as a **conflict requiring manual review**. Possible explanations include data entry errors, identity fraud, or document reuse — each of which is analytically significant.

### 3. Precedence Over Probabilistic Methods
Deterministic matching takes absolute precedence over probabilistic techniques (Task 7.2) because the consequences of error are asymmetric.
*   **False Positive Cost:** In law enforcement contexts, erroneously merging two distinct individuals into a single record (a false positive) can result in wrongful accusation, misattribution of criminal activity, or intelligence contamination. Deterministic matching on unique identifiers virtually eliminates this risk.
*   **False Negative Tolerance:** A false negative — failing to link two records that should be connected — is a less catastrophic error than a false positive. It means the connection is missed temporarily but can be recovered through subsequent probabilistic matching or manual review.
*   **Hierarchy of Trust:** Deterministic keys are trusted above all other matching signals. Even if probabilistic algorithms suggest a merge based on name and address similarity, a mismatch on a deterministic key (e.g., different passport numbers) blocks the merge.

### 4. Handling Identifier Absence
In OSINT contexts, unique identifiers are frequently unavailable. An anonymous social media account has no passport number; a shell company in an opaque jurisdiction may have no accessible registration number.
*   **Default to Probabilistic Pipeline:** When no deterministic keys are available, the record is routed to the probabilistic matching pipeline in Task 7.2. The absence of deterministic keys is itself recorded as metadata — it signals that the entity's identity is less certain and that any subsequent merge carries higher risk.
*   **Partial Key Matching:** Some identifiers may be partially available (e.g., a redacted document showing the last four digits of an account number). Partial keys can be used to narrow the candidate pool but cannot trigger an automatic merge — they are treated as supporting evidence for probabilistic assessment.

### 5. Audit Trail for Deterministic Merges
Every automatic merge executed by deterministic matching must be logged with full transparency.
*   **Merge Record:** The Entity Register records which deterministic key triggered the merge, which source records were combined, and which specific fields were contributed by each source.
*   **Data Lineage Preservation:** The original, pre-merge records must be preserved in the system's audit history. If a deterministic merge is later found to be erroneous (e.g., a fraudulently reused passport number), the merged entity must be separable back into its constituent source records without data loss.
*   **Conflict Log:** Any conflicts detected during deterministic matching (matching keys with conflicting attributes) are logged separately for analyst review, with a priority flag proportionate to the investigative significance of the affected entity.
