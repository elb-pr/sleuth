Task 7.2, **Probabilistic Matching and Human Review**, handles the cases that deterministic matching (Task 6.3) cannot resolve — records that lack unique identifiers but share enough attributes to suggest they may refer to the same real-world entity. This is the dominant matching scenario in OSINT investigations, where subjects rarely present government-issued identification.

The detailed procedural breakdown of this task is as follows:

### 1. Attribute-Level Comparison
When no deterministic key triggers an automatic merge, the probabilistic engine compares records across multiple attributes simultaneously.
*   **Name Matching:** Names are the most frequently available but least reliable matching attribute. The engine applies multiple comparison techniques in parallel: exact string match, Jaro-Winkler distance (effective for typos and character transpositions), Soundex and Metaphone (phonetic matching for names that sound alike but are spelt differently), and transliteration normalisation (for names transliterated from non-Latin scripts where multiple Romanisation conventions exist).
*   **Date of Birth Matching:** Transposed digits (e.g., 12/03/1985 vs 03/12/1985) are the most common date-of-birth discrepancy, particularly across US and European date format conventions. The engine checks both orderings. Partial date matches (matching year and month but not day) receive reduced weight.
*   **Address Matching:** Addresses are normalised for common abbreviations (St./Street, Rd./Road, Apt./Apartment), then compared using string similarity. Geographic proximity (e.g., two addresses within the same postcode) receives a lower but non-zero weight.
*   **Behavioural Attributes:** In digital investigations, behavioural attributes — writing style, posting times, shared network connections, device fingerprints — may supplement traditional demographic matching.

### 2. Rarity-Based Confidence Weighting
The diagnostic value of a match depends on the rarity of the shared attribute in the population.
*   **Name Frequency:** A match on "Mohammed Ali" in a UK dataset is far less diagnostic than a match on "Zanzibar Featherstonehaugh" because the former is vastly more common. The probabilistic engine references population frequency tables to weight name matches accordingly.
*   **Geographic Context:** An address match in a densely populated urban area is less diagnostic than a match in a rural area with few inhabitants. Similarly, a date-of-birth match carries less weight if the shared date is 1 January (a common placeholder in datasets with missing dates).
*   **Composite Rarity:** The engine computes a composite confidence score reflecting the combined rarity of all agreeing attributes. Two common attributes agreeing (common name + common DOB) may produce a lower score than a single rare attribute agreeing (unusual surname).

### 3. Candidate Match Presentation
Records whose composite scores fall between the automatic-merge and automatic-reject thresholds are presented to the analyst for manual adjudication.
*   **Match Summary Card:** For each candidate pair, the analyst receives a structured summary showing: all agreeing fields (highlighted), all disagreeing fields (highlighted), the individual weight of each agreement/disagreement, and the composite confidence score.
*   **Source Context:** The analyst must be able to access the original source documents for both records (via the Source Register links from Task 5.3) to assess whether the shared attributes are genuinely indicative of a match or coincidental.
*   **Decision Options:** The analyst may: **Confirm** the match (merging the records with the analyst's identity recorded), **Reject** the match (recording the reason for rejection so the pair is not re-presented), or **Defer** pending additional collection (creating a new Intelligence Requirement to seek disambiguating evidence).

### 4. The Invisible False Positive Problem
The most dangerous failure mode in probabilistic matching is the "invisible" false positive — a merge that appears correct but conflates two distinct entities.
*   **Father-Son Problem:** Individuals sharing a surname, similar address, and overlapping date ranges (e.g., a father and adult son at the same household) can produce high composite scores despite being different people. Mitigation requires negative weighting: if first names differ and age gaps are large, the merge must be blocked or flagged.
*   **Identity Theft/Fraud:** Two records may share a deterministic key (e.g., a stolen passport number) but belong to different individuals. The probabilistic engine must detect attribute conflicts that suggest fraudulent reuse rather than genuine identity.
*   **Must-Not-Match Rules:** The engine should support explicit exclusion rules — pairs of records that an analyst has determined refer to different entities and must never be merged, regardless of future attribute similarity.

### 5. Audit Trail for Probabilistic Decisions
Every probabilistic merge and rejection must be fully auditable.
*   **Decision Record:** For confirmed matches: which attributes agreed, what weights were assigned, what the composite score was, and which analyst approved the merge. For rejected matches: the reason for rejection.
*   **Reversibility:** All probabilistic merges must be reversible. The pre-merge state of both records must be preserved in the audit history, so that if new evidence reveals a false positive, the merged entity can be separated back into its original constituent records without data loss.
*   **Continuous Re-Evaluation:** As new data arrives throughout the investigation, previously rejected pairs should be re-evaluated against the updated evidence base. A pair that was correctly rejected based on two attributes may become a match when a third, disambiguating attribute emerges.
