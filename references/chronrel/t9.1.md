Task 9.1, **Master Chronology Construction**, creates the temporal backbone of the investigation by aggregating all dated events from the Entity Register and Evidence Register into a single, unified chronological matrix. This master timeline is the primary mechanism for establishing causality, identifying coordination between entities, and exposing temporal anomalies that may indicate deception or incomplete intelligence.

The detailed procedural breakdown of this task is as follows:

### 1. Aggregation of Dated Events
Every event recorded across the investigative database — financial transactions, communications, movements, document filings, meetings, digital activity — must be extracted and compiled into a single chronological sequence.
*   **Source Sweep:** The analyst systematically extracts all Event-type entities from the Entity Register (Phase 3), all timestamped entries from the Task Log (Phase 2), and any temporal data embedded in Subject and Corporate Profiles (Task 8.1). No category of dated evidence is excluded from the master chronology.
*   **Event Record Format:** Each entry in the master chronology carries mandatory fields: Date/Time (UTC), Event Description (purely factual, no analytical interpretation), Entity References (which POLE entities are involved), Source Reference (which source documented this event), Source Confidence (Admiralty 1–6 rating), and Temporal Certainty (Exact, Approximate, Estimated, or Conflicting).
*   **Objective Language:** Event descriptions must be written as neutral factual statements. "Entity A transferred £50,000 to Account B" is acceptable. "Entity A laundered £50,000 through Account B" is an analytical judgment that belongs in Phase 5, not in the chronology.

### 2. UTC Normalisation
Multi-jurisdictional and digital investigations generate timestamps in multiple time zones. All timestamps must be mathematically normalised to **Coordinated Universal Time (UTC)** before sequencing.
*   **Conversion Documentation:** When converting a timestamp from local time to UTC, the original timestamp, the timezone offset applied, and the UTC result must all be recorded. This allows a third party to verify the conversion.
*   **Ambiguous Timezones:** If a source provides a timestamp without specifying a timezone (common in informal communications and social media posts), the analyst must document the assumption used (e.g., "assumed GMT based on subject's known UK residence") in the Temporal Certainty field.
*   **Daylight Saving Time:** DST transitions are a documented source of sequencing errors. The analyst must verify whether a timestamp falls within a DST period and apply the correct offset. System logs that do not adjust for DST must be corrected manually.

### 3. First-Pass Assembly (Raw Chronology)
The initial assembly is purely mechanical — events are sorted by timestamp with no analytical filtering.
*   **No Cherry-Picking:** The raw chronology includes all dated events, regardless of their apparent relevance. Routine events that seem unimportant may gain significance when placed in temporal proximity to other events.
*   **Duplicate Detection:** Events that appear in multiple sources are recorded once, with all contributing sources cited. If the sources provide slightly different timestamps for the same event, this is flagged as a conflict for resolution in Task 9.3.
*   **Temporal Resolution:** Events are sequenced to the highest precision available. Where precise times are known (e.g., 14:37:22 UTC from a system log), they are recorded to that precision. Where only dates are known, events are sequenced by date with intra-day ordering marked as unknown.

### 4. Output: The Master Chronological Matrix
The completed raw chronology is formatted as a structured matrix (table) that serves as the primary temporal reference for all subsequent analysis.
*   **Deliverable Format:** A master table with all events, all entities, all sources, and all confidence ratings — the complete documentary base of the investigation rendered as a temporal sequence.
*   **Cross-Reference Capability:** Each event entry links to the corresponding Entity Register records, Evidence Register items, and Source Register entries, enabling any user of the chronology to drill down into the underlying evidence.
*   **Downstream Dependencies:** The master chronology feeds directly into the ACH Matrix (Phase 5), where temporal patterns and sequences become evidence rows. It also feeds the final briefing (Phase 6), where the chronology may be presented as a summary exhibit.
