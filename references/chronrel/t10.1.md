Task 10.1, **Edge List Generation**, transforms the relationship data stored in the Entity Register into a structured, machine-readable format suitable for network analysis and visualisation. The edge list is the fundamental data structure underlying all link charts, social network analysis, and graph-based reasoning. It is compiled using the **Network Architecture (Edge List) Template**.

The detailed procedural breakdown of this task is as follows:

### 1. Mandatory Edge Fields
Each row in the edge list represents a single relationship between two entities. Four fields are mandatory for every edge:
*   **Source Entity:** The unique identifier (from the Entity Register) of the entity from which the relationship originates. In directional relationships (e.g., "Entity A pays Entity B"), this is the acting entity.
*   **Target Entity:** The unique identifier of the entity at the receiving end of the relationship.
*   **Relationship Type:** A typed label from the controlled vocabulary defined in the POLE schema (Task 6.1). Standard types include `OWNS`, `DIRECTOR_OF`, `EMPLOYS`, `COMMUNICATES_WITH`, `FAMILY_OF`, `LOCATED_AT`, `ASSOCIATED_WITH`, and `PARTY_TO`.
*   **Evidence Citation:** A direct reference to the specific source in the Source Register (Task 5.3) that documents this relationship. An edge without an evidence citation is inadmissible — it represents an unsourced analytical assertion rather than a documented fact.

### 2. Supplementary Edge Metadata
Beyond the four mandatory fields, additional metadata enriches the analytical value of the edge list:
*   **Confidence Level:** The Admiralty credibility rating (1–6) of the evidence supporting the relationship. This metadata directly governs the visual grammar in Task 10.3 — high-confidence edges are rendered as solid lines, low-confidence as dashed.
*   **Date Range:** The temporal window during which the relationship was active (start date, end date, or ongoing). This enables temporal filtering of the network, showing how the structure evolves over time.
*   **Weight/Intensity:** For quantifiable relationships (e.g., number of communications, transaction volumes), a numeric weight captures the intensity of the link. Weighted edges enable more nuanced centrality analysis in Task 10.2.
*   **Directionality:** Whether the relationship is unidirectional (A pays B) or bidirectional (A and B are associates). This distinction affects how network analysis algorithms interpret the graph.

### 3. Extraction from the Entity Register
The edge list is not created from scratch — it is systematically extracted from the typed links already recorded in the Entity Register.
*   **Relationship Sweep:** The analyst exports all relationship links from the Entity Register, filtering out any that lack source attribution (orphan links). Each exported link is formatted into the edge list structure.
*   **Deduplication:** If the same relationship appears in multiple sources, it is recorded as a single edge with all contributing sources cited and the highest corroboration status applied.
*   **Indirect Relationships:** At this stage, only directly evidenced relationships are included. Inferred relationships (e.g., "A probably knows B because they both attended the same event") are excluded from the primary edge list but may be explored through Two-Mode Analysis in Task 10.4.

### 4. Output Format
The edge list must be structured for compatibility with downstream tools.
*   **Tabular Format:** The standard output is a table (CSV, TSV, or Markdown) with one row per edge and columns for each field. This format is directly importable into graph analysis tools (NetworkX, Gephi, i2 Analyst's Notebook).
*   **Graph Exchange Formats:** For more advanced tooling, the edge list may be exported as GEXF (Graph Exchange XML Format) for Gephi or GraphML for interoperability with multiple platforms.
*   **Mermaid.js Syntax:** For text-based environments (including AI-assisted analysis), the edge list can be rendered as Mermaid.js graph syntax, with strict adherence to node naming conventions and line-style rules based on confidence levels.

### 5. Quality Gate
Before network analysis proceeds, the edge list must pass a validation check:
*   **No Unsourced Edges:** Every row must have a populated Evidence Citation field. Edges without sources are deleted or flagged for urgent source attribution.
*   **Entity Register Consistency:** Every Source and Target entity in the edge list must have a corresponding record in the Entity Register. Orphan references (entity IDs that do not resolve to a register entry) indicate data integrity failures.
*   **Relationship Type Validation:** Every Relationship Type must match the controlled vocabulary. Free-text or ad-hoc relationship labels are standardised before analysis to prevent fragmentation.
