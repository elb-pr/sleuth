Task 10.3, **Visual Grammar Application**, governs how the network is rendered as a visual product. The critical principle is that the visualisation must preserve evidentiary integrity — the viewer must be able to distinguish confirmed facts from unverified suspicions at a glance. Without strict visual grammar, a link chart becomes a persuasive illustration rather than an analytical tool.

The detailed procedural breakdown of this task is as follows:

### 1. Node Representation
Each entity in the network is represented as a node whose visual attributes encode its type and status.
*   **Semantic Icons:** Different entity types receive distinct visual representations. The i2 Analyst's Notebook convention uses specific avatars: person silhouettes for individuals, building icons for organisations, map pins for locations, calendar icons for events, vehicle icons for transport, and device icons for communication endpoints. In text-based or Mermaid.js environments, distinct node shapes (rectangles, circles, diamonds) serve the same function.
*   **Node Sizing:** Nodes may be scaled by a quantitative metric — typically degree centrality or transaction volume — so that the most connected or most active entities are visually prominent.
*   **Node Colouring:** Colour encodes analytical categories. Common conventions include colouring nodes by cluster membership (from Task 10.2), by entity type, or by risk status (e.g., red for sanctioned entities, amber for PEPs, green for cleared entities).

### 2. Edge Representation (The Confidence Distinction)
The most critical visual grammar rule is the distinction between confirmed and unconfirmed relationships.
*   **Solid Lines:** Represent confirmed associations — relationships supported by evidence graded 1 (Confirmed) or 2 (Probably True) on the Admiralty credibility axis. These are relationships the investigation is prepared to defend.
*   **Dashed or Dotted Lines:** Represent suspected or low-confidence relationships — evidence graded 3 (Possibly True) or lower, or inferred relationships not yet corroborated. These visual markers communicate epistemic doubt directly to the viewer.
*   **Line Weight:** Edge thickness may encode relationship intensity (e.g., transaction volume, communication frequency). Thicker lines indicate stronger or more frequent connections.
*   **Directional Arrows:** Where relationships are asymmetric (e.g., fund flows, employment), directional arrows indicate the direction of the relationship.
*   **Edge Labels:** Relationship types from the controlled vocabulary (Task 6.1) are displayed as labels on or near the edge.

### 3. Mandatory Legend and Documentation
Every visualisation must include a legend — a key that explains all visual conventions used in the chart.
*   **Legend Contents:** Node shape meanings, node colour meanings, line style meanings (solid = confirmed, dashed = unconfirmed), line weight scale, directional arrow meaning, and temporal scope of the data represented.
*   **Classification Marking:** If the investigation has classification or handling restrictions, the chart carries the appropriate marking.
*   **Date of Production:** The chart is timestamped to indicate when it was generated and what data it reflects. Networks evolve; a chart produced on 15 March may not accurately represent the network on 30 March.

### 4. Preventing the "Hairball" Problem
Dense networks with hundreds of nodes and thousands of edges become visually impenetrable — the "hairball" — providing no analytical insight.
*   **Filtering:** The analyst applies thresholds to display only edges above a minimum confidence level or weight, or only nodes within specific clusters. The full dataset remains in the edge list; the visualisation is a curated analytical view.
*   **Sub-Charts:** Complex networks are segmented into sub-charts by cluster, entity type, time period, or geographic region. Each sub-chart focuses on a specific analytical question.
*   **Layout Algorithms:** Automatic layout algorithms (hierarchical, force-directed, circular) are applied to expose structural patterns. The choice of algorithm is documented, as different layouts can make the same data appear structurally different.

### 5. Output Formats
*   **Interactive HTML:** pyvis (via vis.js) produces interactive, zoomable network visualisations that allow the viewer to explore the network dynamically.
*   **Static Image:** High-resolution PNG or SVG exports for inclusion in reports and briefings.
*   **Mermaid.js:** For text-based AI-assisted environments, strict Mermaid.js graph syntax with coded line styles (solid/dashed) based on confidence levels.
*   **GEXF/GraphML:** For import into Gephi or other analysis platforms for further manipulation.
