Task 5.1, **Digital Evidence Hashing**, is the foundational technical step for maintaining the integrity of the digital chain of custody. Every file, screenshot, or digital artefact collected during the investigation must be processed through a cryptographic hashing algorithm at the moment of capture, creating an immutable "digital fingerprint" that proves the material has not been altered at any subsequent point.

The detailed procedural breakdown of this task is as follows:

### 1. The Cryptographic Hashing Process
A cryptographic hash function takes an input of any size — a screenshot, a PDF, a video file — and produces a fixed-length string of characters (the "hash value" or "digest") that is unique to that specific input.
*   **Algorithm Standard:** The investigation mandates the use of **SHA-256** (Secure Hash Algorithm, 256-bit) as the minimum standard. SHA-256 is widely accepted in legal, regulatory, and intelligence contexts and is computationally infeasible to reverse or forge.
*   **Deterministic Output:** The same file will always produce the same hash value. If even a single bit of the file is altered — whether through deliberate tampering, accidental corruption, or format conversion — the hash value changes completely, making any modification instantly detectable.
*   **Collision Resistance:** SHA-256 is designed so that it is computationally impractical to find two different inputs that produce the same hash value. This ensures that each hash is a reliable unique identifier for its corresponding file.

### 2. When to Hash
Timing is critical. The hash must be generated at the point of collection, not retrospectively.
*   **Immediate Hashing:** The hash is computed the moment the digital file is captured, downloaded, or screenshotted. Any delay between capture and hashing introduces a window during which the file could theoretically be altered, undermining the evidentiary chain.
*   **Automated Hashing Tools:** Where possible, hashing should be automated using tools that compute and record hashes as part of the capture workflow. Hunchly, for example, automatically hashes web pages and screenshots during browsing sessions. For manual workflows, Python's built-in `hashlib` library or command-line tools (`sha256sum` on Linux) are standard.
*   **Re-Hashing on Transfer:** Whenever evidence is moved between storage systems (e.g., from a working directory to a read-only evidence vault), the hash must be recomputed and compared against the original to verify that no corruption occurred during transfer.

### 3. Recording and Storing Hash Values
The hash value itself must be stored separately from the evidence file to prevent a scenario where both are altered simultaneously.
*   **Evidence Register Entry:** Each hash value is recorded as a mandatory metadata field in the **Evidence / Material Register**, alongside the file's unique reference number, filename, file size, format, and the date/time of capture.
*   **Separation of Storage:** The hash log and the evidence files should not reside in the same writable location. Best practice stores the hash log in a separate, access-controlled system or appends it to the **Case Decision Log** as an immutable entry.
*   **Witness Verification:** In high-stakes or litigation contexts, a second individual may independently verify the hash at the time of capture, creating a corroborated record of the file's original state.

### 4. Verification and Audit
Hash values are not a one-time artefact — they are an ongoing verification mechanism throughout the life of the investigation.
*   **Periodic Integrity Checks:** The Phase 2 Administrative Checkpoints mandate periodic verification that all items in the Evidence Register still produce their recorded hash values. Any discrepancy triggers an immediate investigation into the cause.
*   **Court and Disclosure Readiness:** If evidence is submitted for legal proceedings, the hash value provides the mathematical proof that the file presented in court is byte-for-byte identical to the file captured during the investigation. Under the Federal Rules of Evidence (FRE 901, US) and eIDAS 2 (EU), hash-verified digital files are generally accepted as authenticated evidence.
*   **Archival Integrity:** When evidence is moved to read-only archival storage at the conclusion of the investigation (Phase 6, Task 14.3), a final hash verification seals the digital chain of custody.

### 5. Limitations and Complementary Measures
Hashing proves integrity but does not prove authenticity or provenance on its own.
*   **No Authentication:** A hash confirms that a file has not been altered since it was hashed. It does not confirm that the file accurately represents reality. A fabricated document, once hashed, will produce a perfectly valid hash. Authentication requires the complementary verification steps in Task 3.3b (geolocation, chronolocation, metadata analysis).
*   **No Access Control:** Hashing does not prevent unauthorised access to or copying of evidence files. Access control is managed through the evidence vault's permissions and logging systems.
*   **Format Sensitivity:** Converting a file between formats (e.g., PNG to JPEG, or saving a web page as PDF) changes its binary content and therefore its hash. The original format must be preserved alongside any derivatives, and each format variant receives its own hash.
