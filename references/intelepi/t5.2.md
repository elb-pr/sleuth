Task 5.2, **Metadata Preservation and Archiving**, ensures that the contextual information surrounding a digital artefact — when it was created, where, by whom, and under what conditions — is captured and preserved alongside the artefact itself. Where Task 5.1 proves that a file has not been tampered with, Task 5.2 preserves the data needed to understand what the file represents and when it was captured.

The detailed procedural breakdown of this task is as follows:

### 1. EXIF Metadata Extraction
Digital photographs and some video formats contain embedded **Exchangeable Image File Format (EXIF)** data that records the conditions of capture at the hardware level.
*   **Standard EXIF Fields:** Camera make and model, date and time of capture, exposure settings (shutter speed, aperture, ISO), focal length, and — critically — GPS coordinates if the device's location services were active at the time of capture.
*   **Extraction Tools:** EXIF data is extracted using forensic-grade tools such as **ExifTool** (the gold standard, supporting 400+ file types), **PyExifTool** (Python wrapper for scripted extraction), or **exifread** (lightweight Python library). The extracted metadata is exported to a structured format (JSON or CSV) and linked to the corresponding entry in the **Evidence Register**.
*   **Analytical Value:** EXIF GPS coordinates, when present, provide a direct geolocation anchor that can be cross-referenced against the visual geolocation performed in Task 3.3b. Date/time stamps provide a chronological anchor. Camera model identification can link multiple photographs to the same device, establishing attribution chains.

### 2. Social Media Metadata Considerations
Most major social media platforms strip EXIF data from uploaded images and videos as a privacy measure. The investigator must account for this.
*   **Metadata Absence is Not Suspicious:** The absence of EXIF data in social media content is expected behaviour, not an indicator of tampering. The investigator records "EXIF stripped by platform" as a factual observation, not a negative finding.
*   **Platform-Specific Metadata:** While EXIF is stripped, platforms often provide their own metadata — upload timestamps, user account identifiers, engagement metrics, and embed codes. This platform metadata is captured and recorded as a separate data layer.
*   **Original Source Priority:** Where possible, the investigator should attempt to obtain the original, uncompressed file (e.g., through direct contact with the uploader, or by locating the same content on platforms that do not strip metadata). The original file, with intact EXIF data, is the evidentiary gold standard.

### 3. Web Page Archiving
Digital investigations rely heavily on web-based content that is inherently ephemeral — pages are edited, deleted, or moved without notice. Archiving captures the state of web content at the moment of observation.
*   **Automated Capture Tools:** The recommended evidence-grade archiving pipeline uses specialised tools to capture the complete state of a web page:
    *   **Hunchly:** A browser extension that automatically tracks every URL visited during a research session, capturing screenshots, timestamps, and SHA-256 hashes. Purpose-built for OSINT evidence preservation.
    *   **ArchiveBox:** A self-hosted archival tool that saves web pages in multiple formats (WARC, HTML, PDF, PNG) simultaneously, providing redundancy against format-specific failures.
    *   **Playwright:** A browser automation framework that captures full-page screenshots and DOM snapshots, producing evidence-grade PNG files with visible URL bars and timestamps.
*   **Independent Corroboration:** After local capture, the investigator should submit the URL to the **Wayback Machine** via the `waybackpy` Python interface. This creates an independent, third-party-timestamped archive that corroborates the investigator's own capture.
*   **WARC Format:** For ISO 28500-compliant archiving, web content is saved in **WARC (Web ARChive)** format using the `warcio` library. WARC files encapsulate the full HTTP request/response cycle, preserving not just the visible content but the underlying network interaction.

### 4. Chain-of-Custody Metadata
Beyond the content metadata, the investigator must record the circumstances of the collection itself.
*   **Collector Identity:** Who performed the capture.
*   **UTC Timestamp:** When the capture occurred, normalised to UTC to prevent timezone ambiguity.
*   **Collection Method:** Which tool or process was used (e.g., "Hunchly browser extension", "manual screenshot via Firefox", "ArchiveBox CLI capture").
*   **Network Context:** Where relevant, the IP address or VPN exit node used during the collection session. In OSINT operations, this information may be required to demonstrate that the investigator did not use covert or deceptive access methods.
*   **Storage Location:** The file path or evidence vault identifier where the archived material is stored, linked to the SHA-256 hash from Task 5.1.

### 5. Handling the Death of Web Caches
The investigator must be aware of the current state of web caching infrastructure, which has degraded significantly:
*   **Google Cache:** Discontinued in February 2024. No longer available as a verification source.
*   **Bing Cache:** Reportedly discontinued in December 2024.
*   **Remaining Options:** The Wayback Machine is now the primary public cache alternative. Archive.today remains operational but faces reliability concerns and should be used with caution for evidentiary purposes. The evidence-grade archiving pipeline (capture locally, hash, submit to Wayback Machine) is the recommended approach.
