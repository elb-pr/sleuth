Task 5.3, **Maintaining the Source Register**, establishes and manages the parallel catalogue of every source consulted during the investigation. Where the **Evidence Register** (Tasks 5.1 and 5.2) tracks the digital artefacts themselves, the Source Register tracks the origins of those artefacts — creating a complete map of where the investigation's information came from, how reliable each origin is, and what restrictions govern its further use.

The detailed procedural breakdown of this task is as follows:

### 1. Source Register Structure
Each source used during the investigation receives a dedicated entry in the Source Register with the following mandatory fields:
*   **Source Identifier:** A unique reference number (e.g., SRC-001, SRC-002) that links the source to every intelligence record it has contributed to across the Entity Register, Evidence Register, and Chronological Matrix.
*   **Source Type:** Classified according to the intelligence taxonomy: Human (HUMINT — interviews, witness statements), Technical (TECHINT — system logs, intercepted data), Open (OSINT — publicly accessible information), or Liaison (provided by another agency or organisation).
*   **Source Description:** A sufficiently detailed description to enable a third party to locate and re-query the source. For open sources, this includes the full URL, database name, or publication reference. For human sources, the description must balance identifiability with any handling restrictions that protect the source's identity.
*   **Admiralty Reliability Rating (A–F):** The source's reliability grade as assessed in Task 3.1b, with a brief justification for the assigned rating.
*   **Track Record Summary:** For recurring sources, a brief history of the source's reporting accuracy across this and previous investigations.
*   **Potential Motivation Assessment:** What does the source gain from providing this information? This field guards against deliberate disinformation by forcing the analyst to consider the source's incentives.
*   **Access Assessment:** Can the source plausibly know what they claim to know? A factory worker reporting on boardroom decisions, for example, has limited plausible access.

### 2. Linking Sources to Intelligence Records
The Source Register does not exist in isolation — it is the provenance backbone of the entire investigative database.
*   **Bidirectional Referencing:** Every entry in the Entity Register (Phase 3) must reference the Source Identifier that contributed each data point. Conversely, every Source Register entry must list which Entity Register records it has informed. This bidirectional linking ensures that if a source is later discredited, every record it contributed can be immediately identified and reassessed.
*   **Multi-Source Attribution:** When a single data point is corroborated by multiple sources (as mandated by the Parallel Sourcing strategy in Task 3.2b), all contributing source identifiers are recorded against that data point. The highest corroboration status (e.g., "Confirmed by SRC-001, SRC-004, SRC-007") is reflected in the Admiralty credibility rating.
*   **Orphan Record Prevention:** Any intelligence record in the Entity Register that lacks a corresponding Source Register entry is flagged as an "orphan" — data with no documented provenance. Orphan records must either be attributed to a source or removed from the analytical workspace.

### 3. Handling Restrictions and Compartmentalisation
Not all sources can be freely cited or shared. The Source Register tracks the handling constraints that govern each source's information.
*   **UK 5×5×5 Handling Code:** In UK policing contexts, each source is assigned a handling code (1–5) specifying who may receive the information and what actions they may take with it. This third dimension of the 5×5×5 system is recorded in the Source Register and enforced during dissemination (Phase 6).
*   **Source Protection:** Human sources may require anonymisation or compartmentalisation. The Source Register records the sensitivity level, and the **Dissemination Log** (Phase 6, Task 14.2) ensures that sensitive source identities are separated from the main case file (e.g., placed in MG6 schedules in UK law enforcement).
*   **Access Method Restrictions:** Some sources may have been accessed through methods that are not publicly disclosable (e.g., subscription databases, liaison intelligence). The Source Register records these restrictions to prevent inadvertent disclosure during reporting.

### 4. Source Lifecycle Management
Sources are not static — their reliability, availability, and relevance change over time.
*   **Date First Assessed:** When the source was first evaluated and entered into the Register.
*   **Last Reviewed:** The most recent date on which the source's reliability rating and track record were reassessed. Stale source records — those not reviewed within a defined period — are flagged for reassessment.
*   **Status Tracking:** Sources may be Active (currently providing information), Dormant (previously consulted but not currently engaged), Discontinued (no longer available — e.g., a deleted social media account), or Discredited (found to have provided false or unreliable information).
*   **Discreditation Protocol:** If a source is discredited, the analyst must immediately identify all intelligence records that rely on that source (via the bidirectional links) and reassess their credibility ratings. This cascade reassessment is one of the most critical quality-assurance functions of the Source Register.

### 5. Integration with the Final Briefing (Phase 6)
The Source Register directly feeds the **Mandatory Source Summary Statement** required in the final analytical report (Task 13.4).
*   **Source Base Assessment:** The report must summarise the overall quality, diversity, and significant gaps of the source base. This summary is generated by aggregating the reliability ratings and source types from the Source Register.
*   **Single-Source Dependency Warnings:** If any critical analytical judgment rests on a single source, the Source Register provides the data to flag this dependency explicitly in the final briefing.
*   **Archival:** At the conclusion of the investigation, the Source Register is archived alongside the Evidence Register, forming part of the permanent case file subject to the evidence vault standards defined in Task 14.3.
